# clients.dev

An open database of how AI coding clients are configured. See `README.md` for the full overview.

## Project layout

- `configs/<client-id>/` — TOML source of truth (one folder per client, one file per surface).
- `src/lib/clients/schema.ts` — Zod schema for clients and surfaces (the contract).
- `scripts/generate-catalog.ts` — compiles `configs/**` → `src/data/catalog.json` (validated).
- `src/lib/clients/catalog.ts` — loads + re-validates the catalog for the app and API.
- `src/app/` — Next.js App Router pages; `src/app/api/[[...route]]/route.ts` is the Hono API.
- `src/components/` — UI (shadcn/ui in `ui/`, app components alongside).

## Conventions

- Package manager + runtime: **bun**. Use `bun run <script>` and `bunx`.
- The catalog is regenerated by `bun run generate` (also runs in `dev` and `build`).
- **No type casting** (no `as any`, avoid non-null `!`): use narrowing and Zod parsing.
- **No hallucinated config.** Every path, field, and format must be verifiable in the
  client's official documentation. When unsure, mark the surface `status = "partial"`
  and add a `note` rather than inventing details.
- Keep the visual theme aligned with `andrelandgraf/fullstackrecipes` (Tailwind v4 oklch
  tokens in `globals.css`, shadcn new-york/neutral, Geist fonts).

## Adding a client

1. `configs/<id>/client.toml` (metadata) + `<surface>.toml` per supported surface.
2. `bun run generate` to validate, then `bun run dev` to preview `/clients/<id>`.
